这是一份根据我们前面多轮深度技术探讨、逐步推演和迭代，最终固化而成的**《敏捷 DAG 状态机：工业级 AI Agent 规划与执行系统设计方案》**。

本方案的核心理念是**“认知卸载”**——将不确定性的大模型降格为“无状态的流水线工人”，而将状态维护、依赖管理、生命周期和物理安全锁死在“确定性的 Python 宿主”中，从而在根本上解决大模型在软件工程场景下易产生的“死循环、幻觉乱改、代码冲突及高昂 Token 成本”等问题。

## 二、 敏捷评估与工具调用预算机制

为防止 Agent 陷入无意义的循环调用，系统通过敏捷打分限制单次执行的任务复杂度，并强绑定工具调用预算。

### 1. 估算四大维度
大模型在评估任务时，必须通过以下维度生成结构化证据链：
* **影响文件范围 (Scope)**：需要修改的文件数量。
* **依赖层级深度 (Depth)**：跨越的系统层级（如：DB -> ORM -> API -> Test）。
* **确定性 (Certainty)**：代码库中是否有现成的相似范例可以参考。
* **验证复杂度 (Validation)**：单体/集成测试的编写与 Mock 难度。

### 2. 点数与工具预算映射矩阵
宿主根据评估点数，采用对应的执行或拆解策略，并严格限定工具调用预算（Tool-Call Budget）：

| 任务点数 (Points) | 宿主调度策略 | 工具调用预算上限 (Max Tool Calls) | 备注 |
| :--- | :--- | :--- | :--- |
| **-1 分 (Spike)** | 激活探针（Auto-Spike） | **10 次** | **严格只读**，挖掘项目客观事实填入 Fact DB |
| **1~2 分** | 立即执行（Trivial） | **3 次** | 极微任务，单文件修改 |
| **3 分** | 立即执行（Small） | **5 次** | 局部特征开发，含单体测试 |
| **5 分** | 立即执行（Medium） | **15 次** | **单次执行安全上限**，含测试与少量微调 |
| **8 分及以上** | **强行拦截**，打回拆解 | - | 由 `Decomposer` 拆解为 $\le 5$ 分的子任务 |

### 3. 动态预算控制流
在执行过程中，宿主会动态扣减预算。
* **动态注入**：宿主在每次工具返回（Tool Output）的头部动态注入当前剩余预算，警示 Worker。
* **临界通牒**：预算仅剩 1 次时，宿主将向 Worker 下发最后通牒，Worker 此时仅被允许调用 `submit_task_result`（提交成功或主动申报失败），禁止调用其他工具。
* **超限熔断**：一旦调用次数归零，宿主将强制终止 Worker，执行 `git reset --hard` 清理现场，并将该节点标记为 `FAILED` 触发自愈流程。

## 三、 任务拆解与三阶段合理性审查流程

为防止 `Decomposer` 产生错误的拆解路径导致级联失效，所有子任务并入主 DAG 前必须经过三阶段拦截。

### 1. 三阶段审查工作流
* **Stage 1 (静态强校验)**：由宿主 Python 脚本验证 JSON Schema 规范、确保子任务点数全部 $\le 5$ 分、检查基础字段完整度。
* **Stage 2 (架构师语义审查)**：唤醒高阶 `Architect Agent`，基于全局事实（Facts & ADR）对子任务列表进行语义审计，检测是否符合设计标准。
* **Stage 3 (物理依赖预演)**：宿主通过 NetworkX 等图算法对子任务的 `depends_on` 依赖关系进行拓扑排序检测，确保无循环依赖死锁。

### 2. 架构师评审的四大语义准则
* **可解性 (Solvability)**：子任务在划定的读写范围内是否具备工具可行性，是否错配。
* **完备性 (Completeness)**：所有子任务的交付物（Outputs）聚合后是否能够 100% 覆盖初始父任务需求。
* **非冗余性 (Non-redundancy)**：子任务间是否职责重合，是否存在多余、无意义的重复操作。
* **I/O 契约有效性 (I/O Contract Alignment)**：检查下游所需的 inputs 在前置任务的 outputs 中已被正确定义并提供。

### 3. 结构化驳回与退化防死循环机制
若审查不通过，宿主向 `Decomposer` 返回带有明确缺陷信息的结构化 JSON：
```json
{
  "review_status": "REJECTED",
  "failed_criteria": "I/O_CONTRACT_MISMATCH",
  "reason": "子任务 T0.3 需要使用 T0.1 生成的 JWT_SECRET，但在 T0.1 的 outputs 契约中并未提供该变量。",
  "refinement_suggestion": "请修改 T0.1 的输出契约以暴露 JWT_SECRET，或将 T0.1 与 T0.3 合并。"
}
```
系统设置**最大重试次数为 2 次**，若仍失败则直接挂起任务并通知人工介入，防止 Agent 陷入反复拆解的死循环。

## 四、 宏观与微观探索分层机制

系统严格区分宏观和微观两种探索权限，以防止 Agent 无目的漫游浪费 Token。

1. **宏观探索（Spike / -1 牌）── 解决“我在哪”**
   * **特点**：冷启动时触发，处于**严格只读模式**。
   * **输出**：仅提取客观事实（如：Web 框架、依赖文件路径）写入 `Global Facts`。禁止修改代码，不产出具体执行方案。
2. **微观探索（Micro-Exploration / 执行态）── 解决“具体在那一行”**
   * **特点**：在具体的 $\le 5$ 分任务运行中触发，**读写混合模式**。
   * **控制**：受限于任务定义的 `read_allowlist` 和 `write_allowlist`，工具代理拒绝越界访问。

## 五、 并发隔离沙箱与 Git 安全机制

宿主通过物理沙箱隔离及中心化变基队列，支持多个子任务并发执行，且避免代码冲突。

### 1. 临时沙箱挂载 (Sandbox)
宿主在物理层挂载各自独立的临时目录，并基于当前主干检出独立的影子分支（Shadow Branches）：`shadow/T0.1` 和 `shadow/T0.2`。

### 2. 权限沙箱拦截 (Tool Guardrails)
沙箱内的读写工具根据任务定义进行权限校验：
* **只读白名单 (read_allowlist)**：限制微观探索（Micro-Exploration）范围。
* **可写白名单 (write_allowlist)**：限制修改范围，防止并发任务之间的重合冲突。

### 3. 中心化合并管道 (Host Merge Pipeline)
影子分支在沙箱内通过局部测试后，进入合并队列：
* **顺序并入**：队列首位的影子分支采用 Fast-Forward Merge 写入主干。
* **动态变基与二次验证 (Rebase & Re-test)**：排在队列后方的影子分支拉取主干并执行 `git pull --rebase`。变基后，宿主在该沙箱内**重新运行物理测试**。
* **自动退化处理**：若遇到 Git 冲突，宿主锁定主干，唤醒临时的 `Git Resolver Agent` 解决冲突，通过测试后方可提交。

## 六、 3 层提示词与 KV Cache 优化

为提高大模型服务商的 KV Cache 匹配率，宿主在向 Worker 发送请求时，严格遵守以下三层前缀构造法：

```markdown
===========================================================
【Tier 1：静态规则层】(永远不变，缓存命中率 100%)
- 敏捷工作流守则 (System Prompt)
- 工具 Schema 定义 & 边界限制规则
===========================================================
【Tier 2：低频全局黑板层】(随着 Spike 和开发推进缓慢积累，缓存命中率 > 90%)
- Global Facts (客观事实：项目技术栈、数据库类型、入口路由位置等)
- Global ADR (全局架构决策：代码规范、统一使用的依赖库等)
===========================================================
【Tier 3：高频动态切片层】(每次都变，无缓存，控制在 < 500 Token)
- 探索边界限制 (read_allowlist & write_allowlist)
- 局部输入上下文 / 契约设计 (Contract-First Inputs)
- 物理验证指令 (verification.test_command)
- 动态预算追踪 (Remaining Tool Calls)
===========================================================
```

## 七、 整体流转控制与生命周期管理

宿主利用 **Git Commit-as-Checkpoint（Git 提交即检查点）** 设计，将运行状态与物理代码版本强行绑定，实现进程的绝对可控。

### 1. 任务流转与物理检查点
* **宿主维护强类型状态机**：节点状态（PENDING / ESTIMATING / SPIKING / READY / RUNNING / VERIFYING / SUCCESS / BLOCKED / FAILED）和全局状态（IDLE / PLANNING / RUNNING / PAUSED / SUCCESS / ABORTED）。
* **自动快照**：在每个 `READY` 任务发给 Worker 之前，宿主执行一次隐式提交：
  ```bash
  git commit -am "checkpoint_before_task_T0.x" --allow-empty
  ```
  并将该 Commit SHA 记入 `State DB`。

### 2. 中断与挂起场景
* **主动暂停**：用户点击 `Pause`，宿主发送终止信号，等待当前的原子工具调用完成，提交沙箱临时 Commit 后挂起。
* **被动挂起**：当发生规划失败（超过 2 次重试）、逻辑死锁/预算超限、或关键物理冲突时，宿主自动将全局标为 `PAUSED` 并将内存序列化归档，等待人工介入。

### 3. 人工介入与恢复路径 (HITL)
当系统处于 `PAUSED` 状态时，用户有三种路径恢复执行：
* **路径 A（直接恢复）**：用户排除外部因素干扰后，宿主加载 `State DB`，Worker 带着剩余预算从 Checkpoint SHA 继续运行。
* **路径 B（人工越权）**：若部分逻辑模型难以实现，人类直接修改代码，测试通过后发送 `force-done`，宿主提交并标记该节点为 `SUCCESS`，DAG 继续推进。
* **路径 C（修正自愈）**：人类修改 `Fact DB` 或全局配置，宿主检测到变更后重估 DAG，用户点击 `Resume` 继续。

### 4. 流程结束与清场
* **成功闭环 (Success)**：影子分支全部合并，并在宿主环境运行完整集成测试。调用轻量级大模型对比初始需求与最终 `git diff`，自动生成“交付总结报告”，标记 `COMPLETED`。
* **异常终止 (Abort)**：人类发送终止信号。宿主读取初始 Checkpoint SHA，强制删除影子分支并执行 `git reset --hard`，销毁物理沙箱，不留任何脏代码。

## 八、 方案总结与核心技术收益

本方案旨在通过确定性的宿主控制减少大模型幻觉引起的级联错误，在实际工程落地中具有以下设计收益：

1. **确定性高**：利用静态与物理层面的双重限制（NetworkX、Git、只读沙箱、白名单代理），使模型在安全和受控的轨道上运行。
2. **零冲突并发**：通过物理隔离沙箱和中心化变基队列，使得多 Agent 并发开发同一个软件系统成为可能。
3. **成本极其可控**：通过 3-Tier 提示词极限压榨大模型的前缀缓存匹配率；通过工具预算拦截和快速回滚阻断死循环造成的资金损耗。
4. **人工介入友好 (HITL)**：Git-based Checkpoint 与 State DB 将整个系统变成了一个可以随时“暂停、倒行、人工修复、继续推进”的可控实体，兼顾了自动化的效率与人类专家的安全把关。
